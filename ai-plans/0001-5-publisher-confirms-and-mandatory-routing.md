## Rationale

Outbound publishing is currently fire-and-forget: `RabbitMqOutboundTarget<TMessage>.DispatchAsync` awaits `BasicPublishAsync`, which only guarantees the bytes left the client, not that the broker accepted or routed the message. A broker NACK, or a `mandatory` message that the broker cannot route, is silently lost. The `Mandatory(bool)` builder option compounds this: it is wired all the way to `BasicPublishAsync(mandatory: ...)`, but nothing observes the resulting `basic.return`, so today it is a reliability knob that does nothing observable.

These two gaps are one feature at the protocol level. With publisher confirmations enabled, an unroutable `mandatory` message produces a `basic.return` *before* its `basic.ack`, so an honest publish outcome must correlate the return against the confirm for the same publish — a returned-then-acked publish is a failure (routed nowhere), not a success. This slice turns publishing into an awaited delivery outcome: a publish that the broker NACKs or returns surfaces as a deterministic exception and a tagged failure metric, and `Mandatory` finally means something. Connection-level recovery/reconnection remains out of scope; the only reconnect-adjacent concession here is a bounded wait so a confirm that never arrives cannot hang a publish forever.

RabbitMQ.Client v7 makes this tractable without manual sequence-number bookkeeping: a channel created with `PublisherConfirmationsEnabled` and `PublisherConfirmationTrackingEnabled` makes `BasicPublishAsync` await the broker ack and surface both a NACK and a correlated `mandatory` return as a `PublishException`. This slice builds on that rather than registering a separate `BasicReturnAsync` handler. See the official tutorial for the channel-options configuration ([Reliable Publishing with Publisher Confirms](https://www.rabbitmq.com/tutorials/tutorial-seven-dotnet)) and the maintainers' description of the awaited ACK/NACK/return-throws behavior ([rabbitmq-dotnet-client Discussion #1713: Publisher confirmation tracking and exceptions](https://github.com/rabbitmq/rabbitmq-dotnet-client/discussions/1713)).

## Acceptance Criteria

- [ ] Publisher confirmations are an opt-in mode configured per channel group. `RabbitMqChannelGroupDefinition` carries the confirm mode, the `ChannelGroup(...)` builder method exposes it, and a topology-level default applies to implicit per-target channel groups. The default preserves the current fire-and-forget behavior so existing configurations are unchanged.
- [ ] Channels created for a confirms-mode channel group are created with `CreateChannelOptions` enabling `PublisherConfirmationsEnabled` and `PublisherConfirmationTrackingEnabled`; channels for fire-and-forget groups are created as today.
- [ ] In confirms mode, a publish that the broker NACKs or returns (unroutable `mandatory` message) surfaces as a dedicated USF exception out of both `PublishMessageAsync<T>` and `PublishRawAsync`, carrying the target name and a delivery-failure reason (at least `nacked`, `returned`, `timeout`). `OperationCanceledException` continues to propagate unwrapped.
- [ ] The publish-failure path records the existing `usf.outbound.publish.failures` counter with the delivery-failure reason available on the `outcome` tag (or an adjacent reason tag), so NACK vs return vs timeout are distinguishable in telemetry. Fire-and-forget publishes keep their current outcomes.
- [ ] In confirms mode the wait for the broker confirm is bounded: an optional confirm timeout (per channel group, falling back to a topology default) plus the caller's `CancellationToken`. Exceeding it yields the `timeout` delivery-failure reason, not an indefinite hang.
- [ ] `Mandatory` is only meaningful with confirms. Compile-time validation rejects a target whose effective channel group is in fire-and-forget mode but sets `Mandatory(true)`, reported through `OutboundTopologyValidationException` together with the other deterministic validation errors.
- [ ] A NACK or return leaves the channel open, so the channel lease returns the channel to the pool rather than discarding it; a delivery-failure exception does not reduce the group's live-channel count. (Channel close/fault discard behavior is unchanged.)
- [ ] XML doc comments on the confirms mode and `Mandatory` state the per-channel serialization and throughput trade-off of confirmation tracking and its interaction with channel-group size.
- [ ] Automated tests are written, covering: confirms-mode happy path (acked), an unroutable `mandatory` publish surfacing as a `returned` failure, the validation error for `Mandatory` without confirms, the bounded-wait `timeout` reason, and that a returned/nacked channel is reused rather than discarded. Integration tests cover the acked and returned paths end-to-end against RabbitMQ via Testcontainers.

## Technical Details

Confirm mode belongs on the channel group because publisher confirmations are a channel-level property and channel groups already own channel creation, pooling, and lifetime. Add the mode to `RabbitMqChannelGroupDefinition` (e.g. a `RabbitMqPublisherConfirmMode { FireAndForget, Confirms }` plus an optional confirm timeout), surface it on `RabbitMqOutboundTopologyBuilder.ChannelGroup(...)`, and add a topology-level default on `RabbitMqOutboundTopologyBuilder` that the compiler applies to the implicit single-channel group it synthesizes for targets that do not name a group. Thread the resolved mode into `RabbitMqOutboundTopologyConfiguration` so the compiler can act on it.

Channel creation is already abstracted as the `Func<CancellationToken, Task<IChannel>>` factory passed into `RabbitMqChannelGroup`/`DefaultRabbitMqChannelPool`. Keep that seam: the compiler builds the factory closure per group and, for confirms-mode groups, calls `connection.CreateChannelAsync(new CreateChannelOptions(publisherConfirmationsEnabled: true, publisherConfirmationTrackingEnabled: true), ct)` instead of the current optionless call. No change is needed to the pool, lease, or health/discard machinery, which key off `IChannel.IsOpen` and observed `ChannelShutdown` — a NACK or return does not close the channel, so the existing return-to-pool path is already correct; the tests should pin this rather than new code introducing it.

`RabbitMqOutboundTarget<TMessage>.DispatchAsync` stays structurally the same (`await using` lease, then `BasicPublishAsync`). With tracking enabled the awaited `BasicPublishAsync` throws on NACK and on a correlated return, so the only addition is translating the broker's publish exception (e.g. the v7 publish exception whose return flag indicates an unroutable `mandatory` message) into a USF delivery-failure exception that carries the target name and a reason. The bounded wait is a linked `CancellationTokenSource` combining the caller token with the group/topology confirm timeout; a timeout maps to the `timeout` reason. Cancellation from the caller's own token must remain an `OperationCanceledException`, consistent with the existing serializer/publish contract.

Introduce the delivery-failure exception in `Usf.Core.Messaging.Errors` (transport-agnostic, with a reason value the transport populates) so `MessagePublisher` can read the reason for the `outcome`/reason tag in its existing catch-and-record block. The publisher's diagnostics flow does not otherwise change: it already records `publish.failures` on any thrown exception; this slice only enriches the reason dimension. Keep the reason vocabulary small and broker-neutral so a future transport can reuse it.

Validation lives in `RabbitMqOutboundTopologyCompiler`, alongside the existing deterministic checks. After resolving each target's effective channel group (explicit or implicit), reject `Mandatory(true)` against a fire-and-forget group with a sorted, grouped error in the existing `OutboundTopologyValidationException` aggregation. The worst-case channel-budget computation is unaffected in count, but confirmation tracking serializes outstanding confirms per channel, so the technical trade-off (favor a larger channel-group maximum, or accept lower per-channel publish concurrency, when using confirms) should be captured in the mode's XML doc and may inform a higher default size for confirms-mode groups.

Out of scope, by design: connection recovery/reconnection (a confirm lost to a dropped connection is covered only by the bounded timeout here), the wire envelope and message-type header convention, and distributed trace-context propagation. Those are separate concerns that pair with the inbound work, not with delivery confirmation.
