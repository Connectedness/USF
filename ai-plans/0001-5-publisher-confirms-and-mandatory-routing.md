## Rationale

Outbound publishing is currently fire-and-forget: `RabbitMqOutboundTarget<TMessage>.DispatchAsync` awaits `BasicPublishAsync`, which only guarantees the bytes left the client, not that the broker accepted or routed the message. A broker NACK, or a `mandatory` message that the broker cannot route, is silently lost. The `Mandatory(bool)` builder option compounds this: it is wired all the way to `BasicPublishAsync(mandatory: ...)`, but nothing observes the resulting `basic.return`, so today it is a reliability knob that does nothing observable.

These two gaps are one feature at the protocol level. With publisher confirmations enabled, an unroutable `mandatory` message produces a `basic.return` *before* its `basic.ack`, so an honest publish outcome must correlate the return against the confirm for the same publish — a returned-then-acked publish is a failure (routed nowhere), not a success. This slice turns publishing into an awaited delivery outcome: a publish that the broker NACKs or returns surfaces as a deterministic exception and a tagged failure metric, and `Mandatory` finally means something. Connection-level recovery/reconnection remains out of scope; the only reconnect-adjacent concession here is a bounded wait so a confirm that never arrives cannot hang a publish forever.

Confirms are **on by default**. Silent message loss is the worst failure mode a messaging library can have — it surfaces in production as a missing message or a stuck saga, long after the publish appeared to succeed — and this library underpins a saga framework where at-least-once publish is a correctness requirement, not a tuning option. The library is pre-release, so making safe-by-default the default is free now and a breaking throughput regression later. A caller who genuinely tolerates loss (telemetry firehoses, best-effort metrics) opts *out* explicitly with one builder call; the person who wants to trade away safety is the right person to pay for it, not the person who never read the docs. The default channel group stays one private single-channel group per target, so the out-of-box behavior is reliable, order-preserving, and one broker round-trip per publish; throughput is bought deliberately, either by enlarging a channel group (trading away strict per-target ordering) or by opting a route into fire-and-forget. `Mandatory` stays **off by default** even with confirms on, because confirms answer "did the broker accept this?" (a NACK is always a real fault) whereas `Mandatory` answers "did it reach a queue?" — and an unroutable message is legitimately fine in many topologies (fanout/topic exchanges whose consumers are not deployed yet, optional subscribers), so it must remain an explicit opt-in.

RabbitMQ.Client v7 makes this tractable without manual sequence-number bookkeeping: a channel created with `PublisherConfirmationsEnabled` and `PublisherConfirmationTrackingEnabled` makes `BasicPublishAsync` await the broker ack and surface both a NACK and a correlated `mandatory` return as a `PublishException`. This slice builds on that rather than registering a separate `BasicReturnAsync` handler. See the official tutorial for the channel-options configuration ([Reliable Publishing with Publisher Confirms](https://www.rabbitmq.com/tutorials/tutorial-seven-dotnet)) and the maintainers' description of the awaited ACK/NACK/return-throws behavior ([rabbitmq-dotnet-client Discussion #1713: Publisher confirmation tracking and exceptions](https://github.com/rabbitmq/rabbitmq-dotnet-client/discussions/1713)).

## Acceptance Criteria

- [x] Publisher confirmations are a per-channel-group mode that is **on by default**. `RabbitMqChannelGroupDefinition` carries the optional confirm-mode override, the `ChannelGroup(...)` builder method exposes it, and a topology-level default (defaulting to confirms) applies when an explicit or implicit channel group does not override it. Fire-and-forget is the explicit opt-out, per group or via the topology-level default. This is a breaking behavior change from the current fire-and-forget default, which is acceptable pre-release.
- [x] The default channel-group size remains 1 (one private single-channel group per target), so the out-of-box behavior stays order-preserving; turning confirms on does not silently raise the default size. `Mandatory` remains **off by default** even when confirms are on.
- [x] Confirm mode is configured only at channel-group granularity, never per target, because publisher confirmations are fixed on the channel at creation and targets sharing a group share its channels. A target's path to a non-default confirm mode is to be assigned (via `UseChannelGroup`) a channel group with that mode. `Mandatory` stays the per-target knob on `RabbitMqOutboundTargetBuilder<TMessage>`. No per-target confirm-mode setting is added.
- [x] Channels created for a confirms-mode channel group are created with `CreateChannelOptions` enabling `PublisherConfirmationsEnabled` and `PublisherConfirmationTrackingEnabled`; channels for fire-and-forget groups are created as today.
- [x] In confirms mode, a publish that the broker NACKs or returns (unroutable `mandatory` message) surfaces as a single dedicated, sealed exception out of both `PublishMessageAsync<T>` and `PublishRawAsync` — `MessageDeliveryException` in `Usf.Core.Messaging.Errors`, transport-agnostic and paralleling `MessageSerializationException`. It exposes: a `TargetName` (string); a `Reason` of type `MessageDeliveryFailureReason`, an enum with at least `Nacked`, `Returned`, `Timeout`; and the originating broker exception (the v7 `PublishException`) as `InnerException` for the `Nacked`/`Returned` cases (`null` for `Timeout`). It is one exception carrying a reason, not a subtype per reason. `OperationCanceledException` from the caller's own token continues to propagate unwrapped and is never converted to `MessageDeliveryException`.
- [x] The publish-failure path records the existing `usf.outbound.publish.failures` counter with the delivery-failure reason available on the `outcome` tag (or an adjacent reason tag), so NACK vs return vs timeout are distinguishable in telemetry. Fire-and-forget publishes keep their current outcomes.
- [x] In confirms mode the wait for the broker confirm is bounded by a finite default confirm timeout of 30 seconds (per channel group, falling back to a topology default, overridable) combined with the caller's `CancellationToken`. Because confirms are on by default, this default timeout must be finite so "reliable by default" cannot become "hangs by default" if the broker goes silent mid-publish. Exceeding it yields the `timeout` delivery-failure reason.
- [x] `Mandatory` is only meaningful with confirms. Compile-time validation rejects a target whose effective channel group is in fire-and-forget mode but sets `Mandatory(true)`, reported through `OutboundTopologyValidationException` together with the other deterministic validation errors.
- [x] A NACK or return leaves the channel open, so the channel lease returns the channel to the pool rather than discarding it; a delivery-failure exception does not reduce the group's live-channel count. (Channel close/fault discard behavior is unchanged.)
- [x] XML doc comments on the confirms mode and `Mandatory` state the per-channel serialization and throughput trade-off of confirmation tracking and its interaction with channel-group size.
- [x] Automated tests are written, covering: confirms-mode happy path (acked), an unroutable `mandatory` publish surfacing as a `returned` failure, the validation error for `Mandatory` without confirms, the bounded-wait `timeout` reason, and that a returned/nacked channel is reused rather than discarded. Integration tests cover the acked and returned paths end-to-end against RabbitMQ via Testcontainers.

## Technical Details

Confirm mode belongs on the channel group because publisher confirmations are a channel-level property and channel groups already own channel creation, pooling, and lifetime. Add an optional mode override to `RabbitMqChannelGroupDefinition` (e.g. a `RabbitMqPublisherConfirmMode { FireAndForget, Confirms }` plus an optional confirm timeout), surface it on `RabbitMqOutboundTopologyBuilder.ChannelGroup(...)`, and add a topology-level default on `RabbitMqOutboundTopologyBuilder` that the compiler applies whenever an explicit or implicit group does not override it. The topology-level default itself defaults to `Confirms`, so confirms are on without any configuration. Thread the resolved mode into `RabbitMqOutboundTopologyConfiguration` so the compiler can act on it. The compiler synthesizes implicit group names in the reserved `$implicit:` namespace; reject explicit group names that use this prefix so user-defined names cannot collide with internal groups. The finite default confirm timeout lives alongside the mode default and is 30 seconds (chosen to match the conventional default HTTP request timeout); do not pass an infinite/`default` timeout, since with confirms on by default an unbounded wait would let a silent broker stall publishes indefinitely.

Channel creation is already abstracted as the `Func<CancellationToken, Task<IChannel>>` factory passed into `RabbitMqChannelGroup`/`DefaultRabbitMqChannelPool`. Keep that seam: the compiler builds the factory closure per group and, for confirms-mode groups, calls `connection.CreateChannelAsync(new CreateChannelOptions(publisherConfirmationsEnabled: true, publisherConfirmationTrackingEnabled: true), ct)` instead of the current optionless call. No change is needed to the pool, lease, or health/discard machinery, which key off `IChannel.IsOpen` and observed `ChannelShutdown` — a NACK or return does not close the channel, so the existing return-to-pool path is already correct; the tests should pin this rather than new code introducing it.

`RabbitMqOutboundTarget<TMessage>.DispatchAsync` stays structurally the same (`await using` lease, then `BasicPublishAsync`). With tracking enabled the awaited `BasicPublishAsync` throws on NACK and on a correlated return, so the only addition is translating the broker's publish exception (e.g. the v7 publish exception whose return flag indicates an unroutable `mandatory` message) into a USF delivery-failure exception that carries the target name and a reason. The bounded wait is a linked `CancellationTokenSource` combining the caller token with the group/topology confirm timeout; a timeout maps to the `timeout` reason. Cancellation from the caller's own token must remain an `OperationCanceledException`, consistent with the existing serializer/publish contract.

Introduce `MessageDeliveryException` (sealed) and the `MessageDeliveryFailureReason` enum in `Usf.Core.Messaging.Errors`, transport-agnostic. The transport constructs it — translating the v7 `PublishException` (whose return flag distinguishes an unroutable `mandatory` return from a NACK) into `Reason` and passing the `PublishException` as the inner exception, or constructing the `Timeout` case with no inner exception. `MessagePublisher` reads `Reason` for the `outcome`/reason tag in its existing catch-and-record block. The publisher's diagnostics flow does not otherwise change: it already records `publish.failures` on any thrown exception; this slice only enriches the reason dimension. Keep `MessageDeliveryFailureReason` small and broker-neutral (the three values above) so a future transport can reuse the same exception and reason vocabulary; broker-specific detail stays on the inner `PublishException`.

Validation lives in `RabbitMqOutboundTopologyCompiler`, alongside the existing deterministic checks. After resolving each target's effective channel group (explicit or implicit), reject `Mandatory(true)` against a fire-and-forget group with a sorted, grouped error in the existing `OutboundTopologyValidationException` aggregation. The worst-case channel-budget computation is unaffected in count, but confirmation tracking serializes outstanding confirms per channel: with the default size-1 group, confirms-on means each publish waits a broker round-trip before the next on that target proceeds. This is the intended conservative default (correct and ordered, modest throughput). The trade-off — and the two deliberate ways to buy throughput, a larger channel-group maximum (which relaxes strict per-target ordering) or opting a route into fire-and-forget — should be captured in the mode's XML doc. Deliberately do **not** raise the default group size to compensate for confirms; that would silently weaken the per-target ordering guarantee that size-1 exists to provide, trading away a correctness property to address a throughput concern.

Out of scope, by design: connection recovery/reconnection (a confirm lost to a dropped connection is covered only by the bounded timeout here), the wire envelope and message-type header convention, and distributed trace-context propagation. Those are separate concerns that pair with the inbound work, not with delivery confirmation. Also out of scope is the v7 `outstandingPublisherConfirmationsRateLimiter` (`CreateChannelOptions`): with the default size-1 confirms group outstanding confirms per channel are already bounded to one, so the limiter is left unset here and configurable rate limiting can be added in a future plan when larger confirms-mode groups make it useful.
