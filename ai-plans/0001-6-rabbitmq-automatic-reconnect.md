## Rationale

`RabbitMqConnectionProvider` intentionally caches one completed connection task for the outbound topology lifetime. For the first release, USF should make the corresponding recovery strategy explicit: RabbitMQ.Client automatic recovery owns reconnect attempts and restores the cached connection wrapper after transient network failures. USF should require that behavior instead of adding provider-level invalidation on `ConnectionShutdownAsync`, which would compete with the client's recovery loop.

RabbitMQ.Client also restores autorecovering channel wrappers by replacing their inner channels and then raising `IRecoverable.RecoveryAsync`. The channel pool already refuses to lease a channel after observing `ChannelShutdownAsync`, but it treats that observation as permanent. It must recognize the recovery event so an idle pooled wrapper that the client successfully restores becomes leasable again. Automatic recovery is an availability mechanism, not a delivery guarantee: it does not buffer or republish messages, and callers still need retry or an outbox for publishes that fail or time out during an outage.

## Acceptance Criteria

- [x] A RabbitMQ connection factory with `AutomaticRecoveryEnabled == false` is rejected with a clear `OutboundTopologyValidationException`. The check runs when USF first creates the connection, immediately before the client opens it, rather than during topology compilation; this keeps the connection-factory delegate invoked exactly once and consistent with USF's existing lazy connection creation. USF does not silently override the caller's factory configuration.
- [x] `RabbitMqConnectionProvider` continues to cache one successfully created `IConnection` for the topology lifetime and does not invalidate it on `ConnectionShutdownAsync`; RabbitMQ.Client owns transient reconnection.
- [x] A pooled channel that observes `ChannelShutdownAsync` remains unavailable while closed, but becomes healthy and reusable when the same autorecovering wrapper raises `IRecoverable.RecoveryAsync` and reports `IsOpen == true`.
- [x] A channel that closes without recovering is still discarded, decrements the pool's live-channel count exactly once, and can be replaced lazily without leaking a lease slot.
- [x] Pool disposal unsubscribes both shutdown and recovery event handlers before disposing a channel.
- [x] XML documentation on both `RabbitMqOutboundTopologyBuilder.UseConnectionFactory(...)` overloads states that automatic recovery is required, that `NetworkRecoveryInterval` remains caller-configurable, and that `TopologyRecoveryEnabled` remains caller-controlled — RabbitMQ.Client defaults it to `true`, it governs exchange, queue, and binding recovery rather than connection recovery, and it need not be enabled when broker topology is provisioned externally.
- [x] `RabbitMqConnectionProvider` logs connection shutdown, successful recovery, and failed recovery attempts with structured fields while leaving RabbitMQ.Client in control of reconnection. These notifications are diagnostic only: they do not invalidate the cached connection, create a replacement, or start a competing recovery loop.
- [x] A RabbitMQ transport README documents that automatic recovery does not buffer or republish failed or in-flight messages, so applications requiring at-least-once effects must retry safely or use an outbox.
- [x] The RabbitMQ transport README states that RabbitMQ cluster nodes should use a consistent `channel_max`: USF validates the channel budget on the initial connection, while RabbitMQ.Client performs subsequent recovery internally.
- [x] Automated tests are written, including: rejection of a factory with automatic recovery disabled; reuse of the same pooled channel wrapper after shutdown followed by recovery; discard and replacement when shutdown is not followed by recovery; no live-channel-count or lease-slot leak; structured lifecycle logs; and installation and disposal of connection-event subscriptions exactly once.
- [x] An integration test restarts (not recreates) the Testcontainers RabbitMQ broker so the mapped port is preserved for recovery, after a channel has entered a maximum-size-1 pool. It verifies that a publish attempted during the outage fails or times out within a finite bound, waits for RabbitMQ.Client recovery, and verifies that repeated subsequent confirmed publishes succeed without pool exhaustion. Because the in-outage publish acquires and discards the single pooled channel, this test exercises lazy channel replacement and externally observable availability; the same-wrapper-reuse-after-recovery guarantee (criterion above) is owned by the unit tests.

## Technical Details

Keep reconnection ownership inside RabbitMQ.Client. In `RabbitMqOutboundTopologyCompiler.CreateConnectionAsync`, validate the `ConnectionFactory` returned by the configured delegate immediately before calling the client's `CreateConnectionAsync`. If `AutomaticRecoveryEnabled` is false, throw an `OutboundTopologyValidationException` with an actionable message. This validation deliberately runs at first connection creation rather than in `Compile`/`Validate`, because the factory delegate may depend on runtime services or have side effects and should be invoked exactly once. Do not mutate the factory, and do not invalidate the cached connection on `ConnectionShutdownAsync`: RabbitMQ.Client returns an autorecovering `IConnection` wrapper when automatic recovery is enabled, and the provider should keep handing out that stable wrapper across reconnects.

Add structured recovery observability inside `RabbitMqConnectionProvider`, which already creates, caches under its gate, and disposes the single connection and is therefore the natural owner of these subscriptions. Inject an `ILogger` (or `ILoggerFactory`) into the provider — note this changes its public constructor, which is an acceptable breaking change. When the provider first creates the connection, subscribe to `IConnection.ConnectionShutdownAsync`, `RecoverySucceededAsync`, and `ConnectionRecoveryErrorAsync`, logging the connection lifecycle transition and the relevant shutdown reason or recovery exception. These handlers are diagnostic only: they must not clear the provider cache, create a replacement connection, or start a competing recovery loop. Because creation happens once under the gate, the subscriptions are installed exactly once; unsubscribe them before disposing the connection so the disposal-triggered shutdown does not race a diagnostic handler. Apply this unsubscribe-before-dispose ordering in both the asynchronous `DisposeAsync` and the synchronous `Dispose` paths of the provider.

Extend `DefaultRabbitMqChannelPool.PooledChannel` to recognize recoverable channels. When the supplied `IChannel` implements `IRecoverable`, subscribe to `RecoveryAsync` in addition to `ChannelShutdownAsync`. (The concrete `AutorecoveringChannel` returned by the client implements `IRecoverable`, so the runtime check reaches the event. The plain channel implementation also implements the `IRecoverable` marker, so the check is effectively always true and `RecoveryAsync` simply never fires when automatic recovery is off — keep the guard for clarity.) The shutdown handler continues to set `_observedShutdown`; the recovery handler clears it with `Interlocked.Exchange` only after RabbitMQ.Client reports recovery. `IsHealthy` must continue to require both a clear observed-shutdown flag and `IChannel.IsOpen`, so the pool never leases the wrapper during the outage and re-checks liveness even after the flag is cleared. Only connection-outage shutdowns are followed by `RecoveryAsync`; a channel-level fault (for example a 406 `PRECONDITION_FAILED`) raises `ChannelShutdownAsync` without a subsequent recovery, so that wrapper stays unhealthy and is discarded — which is the desired behavior. Unsubscribe both handlers during disposal. A wrapper discarded because a caller acquires during the outage remains discarded; a later acquisition can create a fresh channel after the cached autorecovering connection has recovered. USF adds no publish-timeout logic of its own: during an outage `BasicPublishAsync` and `CreateChannelAsync` raise `AlreadyClosedException` / `OperationInterruptedException`, and the pool already releases the reserved slot on that throw (`DefaultRabbitMqChannelPool.cs:78-82`). The finite bound the integration test asserts is therefore the client's exception behavior — detection governed by the negotiated heartbeat and optionally tightened by the channel group's `PublisherConfirmTimeout` and the caller's `CancellationToken` — not anything USF implements.

Preserve `TopologyRecoveryEnabled` as a caller decision rather than a USF requirement. It defaults to `true` in RabbitMQ.Client and is useful when the client should recreate declared exchanges, queues, and bindings, but deployments that provision durable topology externally may disable it while still relying on connection recovery. Add XML comments on both `RabbitMqOutboundTopologyBuilder.UseConnectionFactory(...)` overloads to document the required and caller-controlled settings. Add `src/Transports/Usf.Transport.RabbitMq/README.md` for the operational guidance: publish behavior during an outage, retry or outbox responsibility, and the requirement that clustered broker nodes use a consistent `channel_max`.

Unit-test recovery behavior with a hand-crafted recoverable test double. The existing `TestRabbitMqChannel` proxies only `IChannel` via `DispatchProxy`, so `is IRecoverable` would be false and the recovery path unreachable; the double must therefore implement both interfaces — for example by proxying a combined `interface ITestRecoverableChannel : IChannel, IRecoverable {}` (or extending the proxy) and adding `add_/remove_RecoveryAsync` handling plus a `RaiseRecoveryAsync()` trigger mirroring the existing `ShutdownAsync()`. It must also be able to transition `IsOpen`. Unit tests should prove reuse of the same pooled wrapper after recovery and verify that lifecycle event subscriptions are installed only once and removed during disposal. For the integration test, configure a short `NetworkRecoveryInterval`, use an explicit maximum-size-1 channel group with confirms enabled (the autorecovering channel reapplies its `CreateChannelOptions`, so confirms survive the reconnect), publish once to populate the pool, restart — rather than recreate — the Testcontainers RabbitMQ broker so the mapped port is preserved, and attempt a publish during the outage, asserting that it faults or times out within a finite bound (a graceful broker stop closes the sockets, so the client detects the outage promptly without waiting a heartbeat). Wait for the client's recovery signal, then perform repeated confirmed publishes. Because the in-outage publish acquires and discards the single pooled channel, this test exercises lazy replacement and proves externally observable availability and bounded failure behavior rather than the identity of RabbitMQ.Client's internal autorecovering wrapper or same-wrapper reuse.
