## Rationale

USF starts a `Producer` activity for every publish (`MessagePublisher.PublishWithDiagnosticsAsync`), tags it, and disposes it — but its `TraceId`/`SpanId` never reach the wire. The span is a dead end: a consumer on the other side of the broker has nothing to link to, so a trace breaks at every hop. For a cloud-native messaging library this is the whole point of the producer span. This slice closes the gap by injecting the producer span's W3C trace context (`traceparent`/`tracestate`, plus baggage) into the published message so the trace actually crosses the broker.

The propagation format is a **transport binding concern**, not a serializer concern — exactly like the `cloudEvents:` header prefix the previous slice 0001-7-cloudevents-serialization.md deliberately kept in the RabbitMQ transport. So injection happens at the RabbitMQ binding layer, using the BCL-native `DistributedContextPropagator` (no OpenTelemetry package dependency), writing **plain, un-prefixed** `traceparent`/`tracestate` headers as the OpenTelemetry messaging semantic conventions specify. This is what broker and consumer auto-instrumentation reads to stitch a consumer span to the producer span. The injection logic is encapsulated once in a public static helper so the automatic path and manual callers share a single implementation.

This is deliberately distinct from the CloudEvents *Distributed Tracing extension* (`cloudEvents:traceparent`), which the CloudEvents spec scopes to *event-creation-time provenance*, not transport-level context propagation. The two are not redundant: the transport header is the trace of *this delivery hop*, whereas the CloudEvents extension is a creation-time snapshot that should survive store-and-forward/replay and therefore belongs with the construction-time, retry-stable attributes (like `id`/`time`). Emitting the CloudEvents tracing extension is a worthwhile, complementary follow-up but is **out of scope here** and deferred to a later slice; this slice only delivers the transport propagation that makes the consumer span link. OpenTelemetry `messaging.*` span semantic attributes (e.g. `messaging.system`, `messaging.destination.name`) are likewise valuable and complementary but deferred to their own slice. The consumer side (extract-on-receive, child/linked span) lands with the deferred inbound work, which will consume what this slice puts on the wire.

## Acceptance Criteria

- [x] When publishing a `CloudEventEnvelope`, the RabbitMQ transport injects the current producer span's W3C trace context into the AMQP application headers as **plain, un-prefixed** `traceparent` (and `tracestate`/baggage when present), readable by standard OpenTelemetry consumer/broker instrumentation. The `cloudEvents:` prefix is **not** applied to these headers.
- [x] Injection honors `System.Diagnostics.DistributedContextPropagator.Current` (which defaults to W3C output) with `Activity.Current` as the source, so the host-configured propagator is respected and no OpenTelemetry package dependency is introduced. When `Activity.Current` is `null` (no active span / no listener), nothing is written and publishing proceeds normally.
- [x] Baggage is propagated when the current activity carries it (the default/W3C propagator emits it); tests assert specific header keys/values rather than an exact header-set equality, so future baggage entries do not break them.
- [x] The injected `traceparent` reflects the `Producer` activity started in `MessagePublisher.PublishWithDiagnosticsAsync` (its `TraceId`/`SpanId`/sampling flags), confirming the publish path keeps that activity current through to the binding. This depends on the activity using the W3C id format (the framework default since .NET 5); under a forced hierarchical id format nothing is injected.
- [x] The trace-context injection logic is encapsulated once in a public static helper (e.g. `TraceContextHeaders.Inject`) with overloads for `IDictionary<string, string?>` (matching `SerializedMessage.Headers`) and `IDictionary<string, object?>` (matching AMQP `BasicProperties.Headers`), both defaulting the activity to `Activity.Current`. The RabbitMQ CloudEvents binding calls this helper, and it is documented as the supported way for raw-publish callers to populate their own headers.
- [x] The automatic raw `PublishRawAsync`/`SerializedMessage` path injects nothing — the escape hatch owns its full wire representation. Callers opt in explicitly via the static helper before constructing the `SerializedMessage`.
- [x] Route headers and the existing `cloudEvents:*` attribute headers are unaffected; if a route header named `traceparent`/`tracestate`/`baggage` ever collides, the propagator-injected value wins (injection runs last).
- [x] XML doc comments note that `traceparent`/`tracestate`/baggage are W3C transport-propagation headers (OpenTelemetry messaging convention), distinct from the CloudEvents attributes and intentionally un-prefixed, and that the CloudEvents Distributed Tracing extension (creation-time provenance) is intentionally deferred to a later slice.
- [x] Automated tests are written: helper unit tests (both overloads; baggage propagated; no-op when no activity is current), a transport unit test asserting the injected `traceparent` matches a known current activity (and `tracestate` round-trips), and an extension of the existing Testcontainers integration test asserting a plain `traceparent` header is present on the wire and recovers the published `TraceId`.

## Technical Details

**Shared helper (`Usf.Core.Messaging`, e.g. `TraceContextHeaders`).** A public static class encapsulates the propagation so the automatic and manual paths share one implementation:

```csharp
public static void Inject(IDictionary<string, object?> headers, Activity? activity = null) =>
    DistributedContextPropagator.Current.Inject(
        activity ?? Activity.Current,
        headers,
        static (carrier, key, value) => ((IDictionary<string, object?>)carrier!)[key] = value);
```

A second overload takes `IDictionary<string, string?>` for `SerializedMessage.Headers` (raw-publish callers); both setters only ever receive `string` values, so both element types are safe. `Inject` is a no-op when the resolved activity is `null`, satisfying the "no active span" criterion for free. The helper lives in `Usf.Core` (which already references `System.Diagnostics.DiagnosticSource` via `Activity`/`ActivitySource`); the W3C header names are transport-neutral, so a future transport reuses the same helper — only the carrier element type differs, which the overloads absorb.

**Injection point (`RabbitMqOutboundTarget.CreateBasicProperties(CloudEventEnvelope, routeHeaders)`).** After the `cloudEvents:*` attribute headers and route headers are populated into the `headers` dictionary, call `TraceContextHeaders.Inject(headers)` as the final step before assigning `properties.Headers`. The CloudEvents binding always allocates a headers dictionary (it always writes `id`/`specversion`/`source`/`type`/`time`), so there is no headerless fast-path to guard. This mirrors the `HttpClient` `DiagnosticsHandler` pattern.

**Why `Activity.Current` is the producer span here.** `OutboundDiagnostics.ActivitySource.StartActivity(…, ActivityKind.Producer)` sets `Activity.Current`, and the publish closure in `PublishWithDiagnosticsAsync` runs within that scope: `→ OutboundTarget<T>.PublishAsync → CloudEventMessageSerializer.SerializeAsync → PublishTypedCloudEventAsync → DispatchAsync → CreateBasicProperties`. The activity is still current at the binding, so no plumbing of the activity/context through method signatures is required. `StartActivity` always creates a *new* span (it never reuses an ongoing one) but only when a listener is registered; if it returns `null` while an ambient application span exists, that ambient span is propagated instead — also correct. With no listener and no ambient span, injection is a clean no-op.

**Why honor `DistributedContextPropagator.Current` rather than hardcode `CreateW3CPropagator()`.** `.Current` already defaults to W3C output, so the modern standard ships by default. Hardcoding a W3C propagator would override a host that deliberately configured B3/composite/pass-through and silently break its chain — the wrong default for a library; honoring `.Current` matches `HttpClient`/ASP.NET Core. Note `DistributedContextPropagator.Current` is a distinct knob from the OpenTelemetry SDK's `Propagators.DefaultTextMapPropagator`; an OTel-SDK-configured propagator does not change the BCL `.Current`. Exact parity would require an `OpenTelemetry.Api` reference, which is intentionally avoided — both default to W3C+baggage, so wire output is identical for the common case. A host wanting strict W3C-only can set `DistributedContextPropagator.Current` itself.

**Why `DistributedContextPropagator`, not hand-formatted headers.** It lives in `System.Diagnostics` (BCL, no extra package), honors the configured propagator, and handles `tracestate`, baggage, and sampling flags correctly. Hand-building `00-{traceId}-{spanId}-{flags}` would hard-code W3C and drop `tracestate`/baggage.

**No core/serializer changes.** `CloudEventMessageSerializer`, `CloudEventEnvelope`, `CloudEventMetadata`, and `MessagePublisher` are untouched: the envelope stays transport-neutral and the propagation format remains a transport-binding detail. The transport header is derived from the current activity at dispatch time, independent of the retry-stable `id`/`time` attributes.

**Deferred, complementary follow-ups.** (1) The CloudEvents Distributed Tracing extension (`cloudEvents:traceparent`/`tracestate`) as creation-time provenance — sourced from construction-time `CloudEventMetadata` and retry-stable, with attribute-name constants added to `Usf.Abstractions`, in the spirit of `id`/`time`. (2) OpenTelemetry `messaging.*` span semantic attributes (`messaging.system=rabbitmq`, `messaging.destination.name`, routing key, message id, body size) and the convention span name/kind, enriched on `Activity.Current` at the binding (same current-activity insight), emitted alongside the existing `usf.outbound.*` tags.

**Testing notes.** Transport/helper unit tests register an `ActivityListener` (sampling `AllData`) on the `Usf.Outbound` source, start an activity, drive the publish through the existing `TestRabbitMqChannel`, and assert `BasicProperties.Headers["traceparent"]`. Re-derive the expected value with `DistributedContextPropagator.Current.Inject`/`ExtractTraceIdAndState` against the same activity rather than string-formatting, so the assertion tracks the configured propagator. The integration test extends the current `cloudEvents:*` header assertions in `RabbitMqPublishingIntegrationTests` with a plain-`traceparent` assertion and a `TraceId` round-trip.
