## Rationale

`Utf8JsonMessageSerializer` is transport-agnostic infrastructure and should live in `Usf.Core`, not in `Usf.Transport.RabbitMq`. Keeping it in the RabbitMQ package creates an avoidable architectural dependency for every future transport. The move is also a good point to make the serializer configurable through caller-provided `JsonSerializerOptions` and to resolve serialization metadata from those options in the same spirit as the `WriteGenericValue<T>` pattern used in Light.PortableResults: prefer declared-type metadata when it correctly covers the runtime value, fall back to runtime-type metadata when it does not, and keep the public API slim by not introducing extra serializer overloads unless they are proven necessary. The test suite should be adjusted at the same time so it exercises the production serializer more often and only keeps small serializer-specific test helpers where they add real value.

Note on framework targets: both `Usf.Core` and `Usf.Transport.RabbitMq` currently target `netstandard2.0`. `System.Text.Json`, `JsonTypeInfo`, and source-generated contexts are all available there, but trimming and NativeAOT guarantees only apply to net6+/net8+ consumers of the package. The metadata-aware path described below is therefore implemented unconditionally on the carrier TFM and provides AOT/trim safety for consumers that target modern .NET; multi-targeting `Usf.Core` is explicitly out of scope for this plan.

## Acceptance Criteria

- [x] `Utf8JsonMessageSerializer` is moved from `Usf.Transport.RabbitMq` into `Usf.Core` under a transport-agnostic namespace, and all transport/tests references are updated to consume the core type.
- [x] The serializer's primary public JSON configuration surface is a constructor that accepts `JsonSerializerOptions`. A parameterless constructor (or an equivalent default instance) is kept so simple DI registrations remain a one-liner; both paths feed through the same metadata-aware code path.
- [x] During serialization, the serializer resolves `JsonTypeInfo` from the configured `JsonSerializerOptions` (declared-type first, runtime-type when the runtime type does not match the declared type and the declared-type metadata does not already cover the polymorphic case, e.g. via `[JsonDerivedType]`). Source-generated metadata supplied through `JsonSerializerOptions.TypeInfoResolver` must drive serialization end-to-end.
- [x] The serializer follows the same general approach as `WriteGenericValue<T>` in Light.PortableResults: declared-type metadata is preferred and runtime-type metadata is only resolved when it is actually needed for correct polymorphic serialization.
- [x] The serializer never silently falls back to reflection-based serialization when required metadata is missing on consumer TFMs where trimming/AOT applies; the missing-metadata case surfaces as a clear, deterministic exception.
- [x] The existing `ArgumentNullException` on a null `message` is preserved and runs before any `message.GetType()` call on the runtime-metadata path.
- [x] `System.Text.Json` is removed from `Usf.Transport.RabbitMq` (no other code in that project uses it) and added to `Usf.Core`.
- [x] Existing public surface stays intact: `IMessageSerializer`, `SerializedMessage`, `Target<T>`, and `RabbitMqPublishRouteBuilder<TMessage>.WithSerializer<TSerializer>()` continue to work after the move. The emitted `SerializedMessage` shape is unchanged (`application/json`, `utf-8`, empty headers by default, no `MessageId`/`CorrelationId`).
- [x] Automated tests cover the moved serializer, option-driven behavior, declared-type serialization, polymorphic serialization that must resolve runtime-type metadata, polymorphic serialization that is already covered by `[JsonDerivedType]` on the declared type (must NOT switch to runtime metadata), the deterministic failure when required metadata is missing, and the affected RabbitMQ wiring after the move.
- [x] Serializer test doubles are reduced where practical: core tests that only need to prove serialization happened use the production JSON serializer plus `RecordingTarget<T>` assertions; metadata-specific transport tests use a small decorator around the production serializer instead of a fully separate `IMessageSerializer` implementation.

## Technical Details

Move `Utf8JsonMessageSerializer` next to the core messaging abstractions in a transport-neutral namespace such as `Usf.Core.Messaging.Serialization`. `IMessageSerializer` and `SerializedMessage` stay in `Usf.Core.Messaging` to avoid an unnecessary churn in the most-referenced API surface; only the concrete JSON serializer gains the `Serialization` sub-namespace.

The public surface is a single class with two constructors: a parameterless one that uses a sensible default `JsonSerializerOptions` instance, and one that accepts caller-supplied `JsonSerializerOptions`. No separate `JsonTypeInfo` overload is added — metadata flows in through `JsonSerializerOptions.TypeInfoResolver`, which is how source-generated contexts are normally wired up.

`SerializeAsync<T>` should follow the same overall algorithm as `WriteGenericValue<T>` in [Light.PortableResults](https://raw.githubusercontent.com/feO2x/Light.PortableResults/refs/heads/main/src/Light.PortableResults/SharedJsonSerialization/Writing/SystemTextJsonWritingExtensions.cs):

1. Null-check `message` first (preserve existing `ArgumentNullException`).
2. Obtain `JsonTypeInfo` for `typeof(T)` from the configured options.
3. If `message.GetType() == typeof(T)`, or if the declared-type `JsonTypeInfo` already advertises the runtime type as a known derived type (i.e. `[JsonDerivedType]` / `JsonPolymorphismOptions`), serialize with the declared-type metadata. Switching to runtime-type metadata in this case would drop the polymorphic type discriminator and is a real correctness bug, not just a perf concern.
4. Otherwise resolve `JsonTypeInfo` for `message.GetType()` from the same options and serialize through that metadata-aware path.
5. If metadata cannot be resolved in either step (no resolver, or resolver returns null for the type), throw a clear exception — do not silently fall back to the reflection-based `JsonSerializer.SerializeToUtf8Bytes<T>(T)` overload.

The route-level serializer abstraction does not change. Existing registrations such as `.WithSerializer<TSerializer>()` continue to work after namespace/package updates. The core serializer continues to emit the same transport-agnostic `SerializedMessage` shape (`application/json`, `utf-8`, empty headers by default) so the RabbitMQ transport only changes by depending on the new core location.

Package references: add `System.Text.Json` to `Usf.Core`, remove it from `Usf.Transport.RabbitMq` (a grep confirms no other code in the transport project depends on it).

The tests are a good place to simplify the serializer story. `RecordingSerializer` is currently used only to prove that serialization happened; those tests can become more sociable by using the production JSON serializer and asserting the `SerializedMessage` captured by `RecordingTarget<T>`. Serializer-specific tests should explicitly cover: serialization when `T` already matches the concrete message type, serialization when `T` is an interface or base type and the serializer must resolve runtime-type metadata, serialization when `T` is a polymorphic base whose declared-type metadata already declares the derived type via `[JsonDerivedType]` (must keep the discriminator and not switch metadata), and a deterministic exception when neither declared-type nor runtime-type metadata is available. `RabbitMqIntegrationSerializer` should be replaced by a thin decorator that wraps the production JSON serializer and adds the few metadata values the integration tests care about, so those tests still verify serializer/header precedence behavior without reimplementing JSON serialization. A small faulting serializer helper may still be warranted for exception-path tests such as `TargetTests`, but the suite should converge on one reusable decorator/delegating pattern instead of several unrelated `IMessageSerializer` implementations.
