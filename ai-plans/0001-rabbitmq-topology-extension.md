## Rationale

The first RabbitMQ publishing slice established the transport-agnostic core abstractions and the simplest exchange-based publishing path, but the RabbitMQ transport model is still too narrow for realistic broker topologies. USF now needs to extend the RabbitMQ-specific publishing model so applications can describe richer exchange, queue, and binding configurations, compile different publishing scenarios into dedicated target types, and support multiple publish routes for the same message type without weakening the transport-agnostic surface of `Usf.Core`. This extension should stay focused on RabbitMQ capabilities supported by the current RabbitMQ .NET client API; in particular, exchange `Internal` remains out of scope for now.

## Acceptance Criteria

- [x] `Usf.Transport.RabbitMq` extends its publish-route configuration model so RabbitMQ publishing scenarios are represented explicitly rather than as a single `exchange + routing key` shape, with distinct route definitions for at least fanout, direct, topic, and headers publishing.
- [x] The RabbitMQ transport compiles different publishing scenarios into different `Target<T>` subclasses so the runtime model reflects the actual RabbitMQ publish semantics instead of using a single generic RabbitMQ target type for every route.
- [x] RabbitMQ publish-route configuration supports both a constant routing key and a routing key derived from the message instance for routing-key-based exchange types, while keeping fanout routes free of unnecessary routing-key configuration.
- [x] The RabbitMQ topology model supports both queue bindings and exchange-to-exchange bindings, including their RabbitMQ-specific arguments and declaration behavior.
- [x] RabbitMQ binding configuration no longer models a passive binding declaration mode that RabbitMQ does not support; bindings use an explicit valid binding-declaration model such as `None` and `Ensure`.
- [x] The RabbitMQ exchange and queue models support the RabbitMQ-specific configuration exposed by the current client surface, using first-class properties where appropriate and broker arguments for extensible settings such as TTL, dead-lettering, alternate exchanges, quorum queues, and similar broker features.
- [x] A single message type can have one default topology route used by `IMessagePublisher` resolution plus additional named RabbitMQ targets registered through `ITargetRegistry`, and startup validation rejects ambiguous default routes and duplicate target names deterministically.
- [x] Headers-based publishing defines how route-level headers and serializer-produced headers are combined deterministically so the builder API, runtime target behavior, and validation all agree on the effective header set used for publishing, with serializer-produced headers taking precedence.
- [x] Startup validation rejects incompatible exchange-type and publish-scenario combinations based on the configured RabbitMQ topology definitions, invalid RabbitMQ entity references, unsupported binding declarations, missing serializers, and any invalid routing-key configuration, and reports all discovered failures deterministically through `MessageTopologyValidationException`.
- [x] RabbitMQ topology provisioning provisions exchanges, queues, queue bindings, and exchange bindings according to the compiled model and continues to fail fast when broker state is incompatible with the configured topology.
- [x] Automated tests are written, including unit tests for builder/compiler validation and integration coverage for direct, topic, fanout, headers, and exchange-to-exchange publishing scenarios.

## Technical Details

`Usf.Core` should remain centered around `Target`, `Target<T>`, `IMessageTopology`, and `ITargetRegistry`. The transport-specific expansion belongs in `Usf.Transport.RabbitMq`, where the current route model should be refactored from a flat `RabbitMqPublishRouteConfiguration` into a small hierarchy of RabbitMQ publish-target definitions. A corresponding builder API should make the RabbitMQ intent explicit, for example through route methods such as `ToFanoutExchange`, `ToDirectExchange`, `ToTopicExchange`, and `ToHeadersExchange`. The route builder should also distinguish between the default route for a message type and additional named routes so `IMessagePublisher` keeps exact-type default resolution while `ITargetRegistry` exposes extra explicitly named targets for the same message type. A message type should have at most one unnamed default route. Any additional route for the same message type must declare a unique target name, and unnamed non-default routes should be rejected during startup validation.

The compiled runtime model should mirror those scenarios. The existing single RabbitMQ target implementation should be split into an abstract RabbitMQ-specific `Target<T>` base that handles shared connection, channel, and property translation concerns, plus concrete target types such as `RabbitMqFanoutTarget<T>`, `RabbitMqDirectTarget<T>`, `RabbitMqTopicTarget<T>`, and `RabbitMqHeadersTarget<T>`. Shared internal inheritance below that level is acceptable where it reflects common mechanics, such as a routing-key-aware base for direct and topic targets, but the compiled runtime model should still preserve the distinct RabbitMQ publish scenarios explicitly. Routing-key-based targets should support both a constant routing key captured at startup and a message-derived routing key delegate compiled from the route configuration. Fanout targets should not require routing-key configuration, and validation should reject meaningless or inconsistent routing settings for a given exchange type. Headers-based targets should treat routing keys as invalid configuration and fail fast with a dedicated validation error or argument exception during startup configuration. They should publish using the transport-agnostic `SerializedMessage.Headers` as the baseline header set and may allow route-level fixed headers to be configured in addition; the merge behavior must be deterministic, with serializer-produced headers overriding route-level headers when the same header key is present. If support for non-standard exchange plugins becomes necessary later, it should be introduced as an explicit extension point only once the desired publish semantics are clear.

The topology model should also be widened beyond queue bindings. Exchange, queue, and binding definition objects should stay immutable runtime definitions compiled during startup, but bindings should become polymorphic so the topology can represent both exchange-to-queue and exchange-to-exchange bindings. The current binding declare mode should be replaced with a concrete model that matches RabbitMQ’s actual capabilities, such as `None` for externally managed bindings and `Ensure` for bindings that USF should create during provisioning, since passive declaration applies to exchanges and queues, not bindings. Exchange and queue definitions should continue to expose their main top-level properties directly and preserve a dictionary of broker arguments for extensibility, which is how features such as dead-letter exchange settings, message TTL, max length, quorum queues, alternate exchanges, delayed exchange arguments, and similar RabbitMQ broker options should be expressed. Exchange `Internal` should remain intentionally unsupported in this plan because the RabbitMQ .NET client API currently used by the transport does not expose exchange declaration support for it.

`RabbitMqMessageTopologyCompiler` should become responsible for validating and compiling this richer model into immutable targets and topology definitions. Validation should remain deterministic and exhaustive. It should detect duplicate exchanges, queues, named targets, and conflicting default routes; ensure all referenced exchanges and queues exist; verify serializers are registered; reject invalid combinations such as fanout publishing against a direct exchange or headers publishing against a topic exchange based on the configured exchange definitions; verify routing-key requirements and routing-key delegate usage; reject routing-key configuration for headers-based targets; validate the configured header merge rules for headers-based targets; and reject invalid binding declarations or exchange-binding references. When broker-owned entities are only referenced rather than fully declared by the configured topology, incompatibilities that cannot be proven from configuration alone may still surface during provisioning or publish-time broker interaction. The compiler should then register one default target per message type in `IMessageTopology` and any additional named targets in `ITargetRegistry`, allowing multiple routes for the same message type without making default publish resolution ambiguous.

`RabbitMqTopologyProvisioner` should provision exchanges and queues as it does today, but it must also provision both queue bindings and exchange bindings from the compiled topology model. Provisioning should continue to run once during startup through `ITopologyProvisioner`, remain idempotent for normal startup, and fail immediately when the broker rejects an incompatible definition. Automated tests should cover the expanded builder and validation rules in unit tests and should add RabbitMQ integration scenarios that prove end-to-end publishing for fanout, direct, topic, and headers exchanges as well as exchange-to-exchange routing, including verification that named targets and default targets resolve as intended.
